# Сценарий 1: Знакомство с лабораторией

**Неделя 1** | Время: ~2 ч

## Цель

Научиться управлять лабой Containerlab, различать mgmt- и data-сеть, находить veth на хосте.

После выполнения необходимо уметь объяснить, где заканчивается инфраструктура управления
стендом и где начинается моделируемая сеть операторского маршрутизатора.

## Что нужно знать заранее

- Контейнер — изолированный Linux userspace с собственным сетевым namespace.
- Интерфейс `eth0` в containerlab обычно используется для management-доступа.
- Интерфейсы `eth1`, `eth2` в этой лаборатории моделируют реальные линковые порты роутеров.
- Команды `ip link`, `ip addr`, `ip route` читают состояние Linux kernel.

Рекомендуемое чтение: [../../docs/theory-foundations.md](../../docs/theory-foundations.md),
разделы 1-3.

## Теория

Вы запустили `containerlab deploy`, видите зелёный статус — но `ping` не идёт. Почему?
Потому что в стенде три независимых плоскости, и зелёный статус проверяет только первую.

В стенде три плоскости:
- **Orchestration** — Containerlab: создаёт контейнеры и veth-пары, формирует топологию на
  уровне сетевых namespace. `containerlab inspect` проверяет именно эту плоскость.
- **Data Plane** — Linux kernel: интерфейсы, адреса, маршруты. Пакеты пересылаются
  через kernel FIB, независимо от того, что показывает control plane.
- **Control Plane** — FRR: IS-IS, BGP, RIB. Строит карту сети и вычисляет маршруты,
  но сам пакеты не двигает.

Management-сеть `eth0` (172.20.20.0/24, IPv4) обеспечивает доступ к контейнерам извне,
но **НЕ** является частью моделируемой сети. Data-интерфейсы `eth1`/`eth2` — это то, что
реально участвует в маршрутизации между r1, r2 и r3.

Главная ловушка для новичка: `ping` через management-сеть работает всегда, потому что
это сеть Docker. Но data-сеть (eth1/eth2, IPv6) — совсем другая. Если инженер
видит "ping ок", а на data-интерфейсах нет адресов — маршрутизация не работает, он просто
не туда смотрит.

| Плоскость | Что это | Где смотреть |
|-----------|---------|--------------|
| Orchestration | Containerlab создаёт контейнеры и veth | `containerlab inspect`, `docker ps` |
| Data Plane | Linux kernel: интерфейсы, адреса, маршруты | `ip link`, `ip -6 addr`, `ip -6 route` |
| Control Plane | FRR строит маршруты и соседства | `vtysh`, процессы `zebra`, `isisd` |

Каждая плоскость проверяется изолированно. Контейнер живой → интерфейс поднят → адрес есть
→ IS-IS сосед Up → маршрут в FIB → ping идёт. Пропустили шаг — не знаете, где обрыв.


## Шаги проверки

### 1. Статус лаборатории

```bash
containerlab inspect -t srv6.yml
```

`containerlab inspect` опрашивает Docker и показывает все контейнеры, относящиеся к топологии
`srv6.yml`: имя контейнера, образ, статус (running/exited), IPv4/IPv6 адреса на management-
интерфейсе и назначенные порты. Это быстрый способ увидеть, все ли узлы запущены и доступны
по mgmt-сети. Зелёная строка `running` рядом с каждым контейнером — минимальный признак жизни.

```bash
containerlab inspect interfaces -t srv6.yml
```

`containerlab inspect interfaces` выводит таблицу data-линков, которую containerlab
сконфигурировал между контейнерами. Для каждого линка показано: узел-источник, его интерфейс
(например `eth1`), узел-назначение, его интерфейс и MAC-адреса обоих концов.
Это подтверждает, что топология развёрнута так, как описано в YAML-файле, и veth-пары
созданы корректно. Ошибка здесь — например, отсутствующий линк — означает, что пакеты
физически не смогут дойти между узлами, даже если маршрутизация настроена верно.

Запишите mgmt IPv4 каждого узла — он понадобится для быстрого доступа.

Смысл проверки: убедиться, что containerlab создал контейнеры и назначил management-адреса.
Это проверка инфраструктуры стенда, а не маршрутизации внутри моделируемой сети.

### 2. Граф топологии

```bash
containerlab graph -t srv6.yml --mermaid
```

`containerlab graph --mermaid` генерирует текстовое описание топологии в формате Mermaid
и сохраняет его в `clab-srv6/graph/srv6.mermaid`. Сам файл `.mermaid` — это текстовый граф:
узлы описаны как прямоугольники, линки — как стрелки между ними с подписями интерфейсов.
Этот файл не требует HTTP-сервера и читается напрямую.

```bash
cat clab-srv6/graph/srv6.mermaid
```

`cat` выводит содержимое `.mermaid`-файла в терминал. Вы увидите текстовую диаграмму,
которую можно вставить в любой Mermaid-рендерер (GitHub, VS Code с плагином, Notion).
Убедитесь, что граф соответствует ожидаемой топологии r1—r2—r3 и все линки на месте.

Опционально запустите HTTP-сервер для интерактивного просмотра:

```bash
containerlab graph -t srv6.yml --srv 0.0.0.0:50080
```

Флаг `--srv` поднимает встроенный HTTP-сервер containerlab на порту 50080. Откройте
`http://localhost:50080` в браузере — увидите интерактивный граф, где можно таскать узлы,
зумить и выделять линки. Сервер работает, пока запущена команда; остановите `Ctrl+C`.

### 3. Обход узлов

На каждом r1, r2, r3:

```bash
docker exec -it clab-srv6-r1 bash
```

`docker exec` запускает новый процесс внутри уже работающего контейнера.
Флаг `-it` нужен, чтобы получить интерактивную сессию с TTY (терминалом):
`-i` подключает stdin, `-t` выделяет псевдотерминал. Без `-it` команды вроде `ip link`
сработают, но вы не сможете использовать `htop`, `less`, или увидеть цветной вывод `vtysh`.
`bash` — это shell, который запускается внутри контейнера; он даёт командную строку
с Linux-окружением контейнера (включая его сетевой namespace).

```bash
ip link
```

`ip link` выводит список всех сетевых интерфейсов внутри контейнера и их состояние.
Колонка `state` покажет `UP` или `DOWN`, колонка `link/ether` — MAC-адрес.
Обратите внимание: `eth0` — это mgmt-интерфейс (172.20.20.x), `eth1` и `eth2` — data-линки.
Если `eth1` в состоянии `DOWN`, значит либо линк не поднят, либо на другом конце veth-пары
контейнер не запущен. `lo` (loopback) — всегда `UNKNOWN` и нужен для локального стека.

```bash
ip -6 addr
```

`ip -6 addr` показывает только IPv6-адреса, назначенные на интерфейсы. Флаг `-6`
фильтрует вывод — без него `ip addr` показал бы и IPv4, что создало бы шум.
В выводе вы увидите адреса вида `2001:db8:12::1/64` с пометкой `scope global` (доступны
снаружи контейнера по data-сети) и `fe80::... scope link` (link-local, видны только соседи
на том же L2-сегменте). Проверьте, что адрес на каждом `eth1`/`eth2` соответствует
запланированному в `srv6.yml`.

```bash
ip -6 route
```

`ip -6 route` выводит IPv6-таблицу маршрутизации (FIB — Forwarding Information Base).
Это именно то, чем руководствуется Linux kernel при принятии решения «куда отправить пакет».
В начале будут connected-маршруты (созданы автоматически при назначении адреса на интерфейс),
затем — маршруты, полученные от FRR через IS-IS. Если пакет до `2001:db8:23::3` должен идти
через `eth1`, но в FIB нет соответствующей записи — пакет будет отброшен. Именно здесь
становится видна разница между Control Plane (FRR посчитал маршрут) и Data Plane (kernel
действительно установил его в FIB).

```bash
ps aux | grep frr
```

`ps aux` выводит все процессы контейнера (a — все пользователи, u — user-ориентированный
формат с именем владельца и потреблением CPU/памяти, x — процессы без терминала).
`grep frr` фильтрует только строки, содержащие «frr». Вы должны увидеть:
- `watchfrr` — сторожевой процесс, перезапускает остальные демоны FRR при падении;
- `zebra` — демон-посредник: принимает маршруты от протокольных демонов и устанавливает
  их в kernel FIB через netlink-сокет;
- `isisd` — демон IS-IS: обменивается Hello-пакетами, строит соседства и рассчитывает
  маршруты по алгоритму SPF (Shortest Path First).
Если какого-то из этих процессов нет — соответствующий компонент control plane не работает.

### 4. veth на хосте

```bash
ip link | grep -E 'veth|clab'
```

Эта команда выполняется на хосте (в вашем терминале, не внутри контейнера).
`ip link` на хосте показывает все интерфейсы хост-машины. `grep -E 'veth|clab'` фильтрует
строки, содержащие либо `veth` (veth-пары, созданные Docker/Containerlab), либо `clab`
(интерфейсы, принадлежащие бриджу clab). Каждый veth-интерфейс, который вы видите — это
один конец пары, другой конец находится в сетевом namespace контейнера и виден там как
`eth1` или `eth2`. Сопоставление «veth на хосте ↔ eth в контейнере» — ключ к пониманию,
как containerlab физически соединяет узлы без использования виртуальных коммутаторов
для point-to-point-линков.

```bash
brctl show 2>/dev/null || ip link show type bridge
```

Management-сеть `172.20.20.0/24` реализована через Linux bridge. `brctl show` — устаревшая
утилита из пакета `bridge-utils`, которая показывает все бриджи и подключённые к ним
интерфейсы. `2>/dev/null` прячет ошибку, если `brctl` не установлен. `||` означает:
если `brctl show` завершился с ошибкой, выполни `ip link show type bridge` — современный
аналог, который делает то же самое средствами `iproute2`. Вы увидите бридж `br-<hex>`
(например `br-1720200000`), к которому подключены veth-интерфейсы всех контейнеров.
Именно этот бридж объединяет `eth0` всех узлов в одну L2-сеть и обеспечивает mgmt-доступ.

### 5. Traceroute r1 → r3

```bash
docker exec clab-srv6-r1 traceroute6 -n 2001:db8:23::3
```

`traceroute6` отправляет UDP-пакеты с последовательно увеличивающимся Hop Limit
(TTL в IPv4, Hop Limit в IPv6). Первый пакет уходит с Hop Limit=1 — первый маршрутизатор
отбрасывает его и отвечает ICMPv6 Time Exceeded, раскрывая свой адрес. Следующий — с
Hop Limit=2, и так далее, пока пакет не дойдёт до цели или не исчерпает лимит попыток.
Флаг `-n` отключает DNS-резолвинг (без него traceroute пытается разрешить имена и подвисает).
Адрес `2001:db8:23::3` — это loopback-адрес r3, анонсированный в IS-IS, поэтому маршрут
до него должен быть известен.

Ожидаемый вывод показывает путь r1 → r2 → r3 с указанием адресов входящих интерфейсов
и времени отклика: первый хоп — `2001:db8:12::2` (адрес r2 на линке r1–r2), второй хоп —
`2001:db8:23::3` (loopback r3).

IS-IS соседи на r1:

```bash
docker exec clab-srv6-r1 vtysh -c "show isis neighbor"
```

`vtysh` — это unified shell для FRR, единый интерфейс ко всем демонам (zebra, isisd,
bgpd, etc.). Флаг `-c` передаёт команду и сразу выводит результат, интерактивный режим
не нужен. `show isis neighbor` запрашивает у `isisd` таблицу соседей IS-IS: для каждого
соседа показан интерфейс, состояние (Up/Down/Init), System ID и приоритет.
Если сосед в состоянии `Up` — значит IS-IS Hello-пакеты проходят между r1 и r2, контрольная
плоскость IS-IS функционирует. Если `State: Init` — Hello уходит, но ответ не получен
(проблема на обратном пути или MTU mismatch). Если соседа нет вообще — Hello не доходит
(линк down, адрес не назначен, или firewall).

## Критерии валидации

- [ ] Объяснить разницу `172.20.20.0/24` (mgmt) и `2001:db8:x::/64` (data)
- [ ] Нарисовать топологию r1—r2—r3 с адресами
- [ ] Показать, что FRR запущен (`watchfrr`, `zebra`, `isisd`)
