# ЛР1: Знакомство с лабораторией

**Неделя 1** | Время: ~2 ч

## Цель

Научиться управлять лабой Containerlab, различать mgmt- и data-сеть, находить veth на хосте.

После выполнения студент должен уметь объяснить, где заканчивается инфраструктура управления
стендом и где начинается моделируемая сеть операторского маршрутизатора.

## Что нужно знать заранее

- Контейнер — изолированный Linux userspace с собственным сетевым namespace.
- Интерфейс `eth0` в containerlab обычно используется для management-доступа.
- Интерфейсы `eth1`, `eth2` в этой лаборатории моделируют реальные линковые порты роутеров.
- Команды `ip link`, `ip addr`, `ip route` читают состояние Linux kernel.

Рекомендуемое чтение: [../../docs/theory-foundations.md](../../docs/theory-foundations.md),
разделы 1-3.

## Теория

Вы запустили `containerlab deploy`, видите зелёный статус — но `ping` не идёт. Почему?
Потому что в лаборатории три независимых плоскости, и зелёный статус проверяет только первую.

В лаборатории три плоскости:
- **Orchestration** — Containerlab: создаёт контейнеры и veth-пары, формирует топологию на
  уровне сетевых namespace. `containerlab inspect` проверяет именно эту плоскость.
- **Data Plane** — Linux kernel: интерфейсы, адреса, маршруты. Пакеты пересылаются
  через kernel FIB, независимо от того, что показывает control plane.
- **Control Plane** — FRR: IS-IS, BGP, RIB. Строит карту сети и вычисляет маршруты,
  но сам пакеты не двигает.

Management-сеть `eth0` (172.20.20.0/24, IPv4) обеспечивает доступ к контейнерам извне,
но **НЕ** является частью моделируемой сети. Data-интерфейсы `eth1`/`eth2` — это то, что
реально участвует в маршрутизации между r1, r2 и r3.

Главная ловушка для новичка: `ping` через management-сеть работает всегда, потому что
это сеть Docker. Но data-сеть (eth1/eth2, IPv6) — совсем другая. Если студент
видит "ping ок", а на data-интерфейсах нет адресов — маршрутизация не работает, он просто
не туда смотрит.

| Плоскость | Что это | Где смотреть |
|-----------|---------|--------------|
| Orchestration | Containerlab создаёт контейнеры и veth | `containerlab inspect`, `docker ps` |
| Data Plane | Linux kernel: интерфейсы, адреса, маршруты | `ip link`, `ip -6 addr`, `ip -6 route` |
| Control Plane | FRR строит маршруты и соседства | `vtysh`, процессы `zebra`, `isisd` |

Каждая плоскость проверяется изолированно. Контейнер живой → интерфейс поднят → адрес есть
→ IS-IS сосед Up → маршрут в FIB → ping идёт. Пропустили шаг — не знаете, где обрыв.
## Предусловия

```bash
make deploy
```

## Задания

### 1. Статус лаборатории

```bash
containerlab inspect -t srv6.yml
containerlab inspect interfaces -t srv6.yml
```

Запишите mgmt IPv4 каждого узла.

Смысл проверки: убедиться, что containerlab создал контейнеры и назначил management-адреса.
Это проверка инфраструктуры стенда, а не маршрутизации внутри учебной сети.

### 2. Граф топологии

```bash
containerlab graph -t srv6.yml --mermaid
cat clab-srv6/graph/srv6.mermaid
```

Опционально запустите HTTP-сервер:

```bash
containerlab graph -t srv6.yml --srv 0.0.0.0:50080
```

### 3. Обход узлов

На каждом r1, r2, r3:

```bash
docker exec -it clab-srv6-r1 bash
ip link
ip -6 addr
ip -6 route
ps aux | grep frr
```

### 4. veth на хосте

```bash
ip link | grep -E 'veth|clab'
brctl show 2>/dev/null || ip link show type bridge
```

Найдите bridge mgmt-сети `br-*` (172.20.20.0/24).

Обратите внимание: veth-пары имеют два конца. Один конец находится в namespace контейнера и
виден как `eth1`/`eth2`, второй конец находится на хосте и виден как veth-интерфейс. Containerlab
соединяет эти пары так, чтобы получить заданную топологию.

### 5. Traceroute r1 → r3

```bash
docker exec clab-srv6-r1 traceroute6 -n 2001:db8:23::3
```

## Expected output

```
traceroute to 2001:db8:23::3, 30 hops max
 1  2001:db8:12::2  ...    # r2
 2  2001:db8:23::3  ...    # r3
```

IS-IS соседи на r1:

```bash
docker exec clab-srv6-r1 vtysh -c "show isis neighbor"
# State: Up, Interface: eth1
```

## Критерий успеха

- [ ] Объяснить разницу `172.20.20.0/24` (mgmt) и `2001:db8:x::/64` (data)
- [ ] Нарисовать топологию r1—r2—r3 с адресами
- [ ] Показать, что FRR запущен (`watchfrr`, `zebra`, `isisd`)

## Контрольные вопросы

1. Почему успешный доступ по `172.20.20.x` не доказывает работу маршрутизации r1-r2-r3?
2. Какие интерфейсы относятся к management-plane, а какие к data-plane?
3. Что создаёт containerlab: маршруты, контейнеры, veth-пары или протокол IS-IS?
4. Почему FRR находится в userspace, а маршруты реально применяются в Linux kernel?

## Требования к отчёту

- Схема r1-r2-r3 с data-адресами и отдельно management-адресами.
- Таблица: узел, контейнер, mgmt IPv4, data IPv6, роль интерфейса.
- Короткое объяснение, какие команды проверяли containerlab, Linux kernel и FRR.
