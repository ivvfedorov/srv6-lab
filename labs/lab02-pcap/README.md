# Сценарий 2: Захват и разбор пакетов

**Неделя 1** | Время: ~2 ч

## Цель

Установить tcpdump, захватить ICMPv6, разобрать Ethernet/IPv6/ICMPv6 в Wireshark.

После выполнения необходимо уметь доказать путь пакета не только через `ping`, но и через
пакетный захват на конкретном интерфейсе.

## Что нужно знать заранее

- Ethernet-кадр доставляется только в пределах одного L2-сегмента.
- IPv6-пакет может пройти через несколько маршрутизаторов, сохраняя конечный destination.
- ICMPv6 Echo Request/Echo Reply используется командой `ping6`.
- `tcpdump` видит пакеты на выбранном интерфейсе, а не “во всей сети сразу”.

Рекомендуемое чтение: [../../docs/theory-foundations.md](../../docs/theory-foundations.md),
разделы 2-3.

## Теория

FRR говорит «всё ок», ping идёт — но где доказательство, что пакет действительно
прошёл через r2, а не ушёл коротким путём по management-сети? Единственный судья — tcpdump:
он показывает пакет таким, каким его реально видит интерфейс.

Пакет разбирается по уровням:
- **Ethernet-кадр** (L2) — показывает next-hop MAC на конкретном линке. На каждом
  hop'е source и destination MAC перезаписываются: это адреса текущего и следующего
  узла, а не конечного отправителя и получателя.
- **IPv6-заголовок** (L3) — показывает сквозной путь. Source и Destination **НЕ**
  меняются на transit-узлах — классическая модель hop-by-hop forwarding.
- **ICMPv6** (L4) — полезная нагрузка: Echo Request (Type 128) или Reply (Type 129).

```text
[Ethernet: Src=r1_MAC Dst=r2_MAC] [IPv6: Src=r1 Dst=r3] [ICMPv6: Echo Request]
```

Если захват сделан не на том интерфейсе — вы видите только management-трафик (eth0),
а data-сеть остаётся невидимой. Самая частая ошибка: `tcpdump -ni eth0` на r2 при
ping r1→r3 — пакет идёт через eth1/eth2, а на eth0 его просто нет.

| Уровень | Что смотреть | Почему важно |
|---------|--------------|--------------|
| Ethernet | Src/Dst MAC, EtherType `0x86dd` | Показывает соседей на конкретном линке |
| IPv6 | Source, Destination, Hop Limit, Next Header | Показывает L3-путь и уменьшение Hop Limit |
| ICMPv6 | Type 128/129, identifier, sequence | Показывает Echo Request или Echo Reply |

## Шаги проверки

### 1. Установка tcpdump

```bash
for n in r1 r2 r3; do
  docker exec clab-srv6-$n apk add --no-cache tcpdump
done
```

`for n in r1 r2 r3; do ... done` — bash-цикл, который выполняет команду для каждого
из трёх узлов. `docker exec clab-srv6-$n` подставляет имя контейнера (`clab-srv6-r1`,
`clab-srv6-r2`, `clab-srv6-r3`).

FRR-образ `frrouting/frr:v8.4.0` основан на Alpine Linux, поэтому используется `apk`,
а не `apt`. Флаг `--no-cache` говорит apk не сохранять скачанные пакеты в локальный
кэш (внутри контейнера он не нужен и только занимает место).

Пояснение: базовый FRR-образ минимальный, поэтому инструменты анализа пакетов ставятся
отдельно. Это нормальная практика для лабораторной среды, но в production-среде установка
пакетов на сетевые узлы обычно контролируется отдельно.

### 2. Захват на r1 (линк r1—r2)

Терминал 1:

```bash
docker exec clab-srv6-r1 tcpdump -ni eth1 -w /tmp/lab02.pcap icmp6
```

`tcpdump` — анализатор пакетов, захватывающий трафик с заданного интерфейса.
Флаги:
- `-n` — не резолвить IP-адреса в DNS-имена (быстрее, не создаёт лишних запросов);
- `-i eth1` — слушать интерфейс `eth1` (data-линк r1↔r2). Именно выбор правильного
  интерфейса отделяет полезный захват от бесполезного;
- `-w /tmp/lab02.pcap` — записать пакеты в файл в формате pcap (читается Wireshark);
- `icmp6` — беркли-фильтр (BPF), захватывать только ICMPv6-трафик.

Без фильтра `icmp6` в захват попадёт IS-IS-трафик (Hello-пакеты, LSP-обновления),
что создаст шум.

Терминал 2:

```bash
docker exec clab-srv6-r1 ping6 -c 5 2001:db8:12::2
```

`ping6` — утилита для отправки ICMPv6 Echo Request. Флаг `-c 5` отправляет ровно
5 пакетов и завершается (без `-c` ping идёт бесконечно). Адрес `2001:db8:12::2` —
data-адрес r2 на линке r1↔r2. Если ping успешен, tcpdump на `eth1` гарантированно
увидит и запрос, и ответ.

Остановите tcpdump (`Ctrl+C`), скопируйте файл на хост:

```bash
docker cp clab-srv6-r1:/tmp/lab02.pcap ~/lab02.pcap
```

`docker cp` копирует файл из контейнера на хост (или обратно). Первый аргумент —
`контейнер:путь`, второй — путь на хосте. `~/lab02.pcap` сохранит файл в домашнюю
директорию.

### 3. Разбор в Wireshark

Откройте `lab02.pcap`. Запишите для одного Echo Request:

| Поле | Значение |
|------|----------|
| Src MAC | |
| Dst MAC | |
| Src IPv6 | |
| Dst IPv6 | |
| Hop Limit | |
| ICMP Type | 128 (Echo Request) |

Дополнительно запишите:

| Поле | Вопрос |
|------|--------|
| Ethernet Type | Почему это IPv6? |
| Next Header | Почему это ICMPv6? |
| ICMP Code | Почему он равен 0 для Echo Request? |
| Frame length | Сколько байт занимает пакет на линке? |

### 4. Transit capture

Повторите захват на r2 eth2 при ping r1 → r3:

```bash
docker exec clab-srv6-r2 tcpdump -ni eth2 -c 10 icmp6 &
```

`-c 10` — остановиться после 10 пакетов (не нужно нажимать `Ctrl+C`). Символ `&` в конце
запускает процесс в фоне, освобождая терминал для следующей команды.

```bash
docker exec clab-srv6-r1 ping6 -c 3 2001:db8:23::3
```

Адрес `2001:db8:23::3` — loopback r3, анонсированный в IS-IS. Пакет от r1 пройдёт
через r2 (на eth1 входит, с eth2 выходит) и достигнет r3. tcpdump на r2 eth2 увидит
оба направления (Echo Request и Echo Reply), но с разными MAC-адресами, потому что
MAC перезаписывается на каждом L2-сегменте.

## Критерии валидации

- [ ] pcap содержит ICMPv6 Echo Request и Echo Reply
- [ ] Объяснить, почему на r2 виден transit-трафик r1↔r3
- [ ] Указать разницу между eth0 (mgmt) и eth1/eth2 (data)

## Контрольные вопросы

1. Почему MAC-адреса меняются от линка к линку, а IPv6 destination остаётся конечным?
2. Что произойдёт с Hop Limit при прохождении r2?
3. Почему tcpdump на `eth0` не подходит для доказательства работы data-сети?
4. Чем ICMPv6 Echo Request отличается от Echo Reply в поле Type?

## Артефакты диагностики

- Скриншот или текстовая таблица полей одного Echo Request из Wireshark.
- Объяснение, почему выбран именно интерфейс захвата.
- Сравнение локального захвата r1-r2 и transit-захвата r1-r3 через r2.
