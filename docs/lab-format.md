# Формат лабораторной работы

Этот формат используется для всех ЛР в репозитории. Его задача — сделать занятие
повторяемым и академически проверяемым: студент понимает теорию, формулирует гипотезу,
запускает стенд, получает наблюдаемый результат и фиксирует выводы.

## Структура README лабораторной

1. **Цель** — один практический навык и один теоретический результат.
2. **Что нужно знать заранее** — минимальные понятия, без которых ЛР превращается в копирование.
3. **Теория** — модель: термины, причинно-следственная связь, что смотреть в выводе.
4. **Предусловия** — состояние стенда и команды проверки перед стартом.
5. **Задания** — небольшие шаги от наблюдения к изменению конфигурации.
6. **Expected output** — пример нормального вывода, не полный лог.
7. **Контрольные вопросы** — проверка понимания механизма, а не памяти команд.
8. **Критерий успеха** — проверяемые результаты и минимальные требования к отчёту.

## Учебный цикл

Используйте один и тот же цикл для каждой ЛР:

```text
Прочитать теорию -> сформулировать ожидание -> выполнить команду
-> сравнить с expected output -> объяснить расхождение
-> исправить или зафиксировать вывод.
```

Такой порядок важен: в сетевых лабораториях команда сама по себе ничего не доказывает.
Доказательство — это связь между control plane, data plane и наблюдаемым трафиком.

## Минимальная модель проверки

Для каждой сетевой функции студент должен показать три уровня:

| Уровень | Что доказывает | Типовые команды |
|---------|----------------|-----------------|
| Control plane | Протокол построил нужное состояние | `show isis neighbor`, `show bgp summary` |
| Kernel dataplane | Linux получил маршрут/SID/VRF | `ip -6 route`, `ip link`, `ip rule` |
| Packet evidence | Реальный пакет прошёл или не прошёл | `ping6`, `traceroute6`, `tcpdump`, pcap |

Если в отчёте есть только ping, отчёт считается неполным: ping показывает симптом, но не
объясняет механизм.

## Минимальный формат отчета

```markdown
# Отчет по ЛР<N>

## 1. Цель
Коротко: что проверялось и какой механизм изучался.

## 2. Схема
Топология, адреса, роли узлов, какой режим стенда использовался: base или SRv6.

## 3. Ход работы
Для каждого шага: ожидание -> команда -> важный фрагмент вывода -> вывод.

## 4. Проверка
Ping/traceroute/show/tcpdump/pcap, которые подтверждают результат.

## 5. Ответы на контрольные вопросы
Только по сути: причина, механизм, где это видно.

## 6. Ошибки и исправления
Симптом -> root cause -> fix.

## 7. Итоговая модель
Короткая схема: как изученный механизм проходит путь от протокола к kernel и пакету.
```

## Правила хорошей лабораторной

- Один шаг меняет одну вещь: так проще найти причину ошибки.
- В каждом блоке должен быть наблюдаемый артефакт: маршрут, соседство, SID, pcap, лог.
- Любой `show` нужно связывать с уровнем системы: FRR RIB, kernel FIB, packet capture.
- Раздел `Expected output` показывает ориентир, но студент должен объяснить свой реальный вывод.
- В конце ЛР всегда должна быть очистка или понятное состояние для следующей ЛР.
- Команды из шпаргалки можно использовать, но в отчёте нужно объяснять не команду, а наблюдаемый
  сетевой механизм.

## Быстрые проверки перед занятием

```bash
make deploy
make verify
```

Для SRv6-блока:

```bash
make srv6
docker exec clab-srv6-r1 vtysh -c "show segment-routing srv6 locator"
```

Для VPN-блока:

```bash
make vpn
docker exec clab-srv6-r1 vtysh -c "show bgp summary"
```
